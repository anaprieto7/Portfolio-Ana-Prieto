WEMALO WMS — Interface Redesign for the Warehouse Management System

Complete visual redesign of the WEMALO warehouse management system, modernizing a dense, enterprise interface while keeping the underlying functionality and tech stack.

SaaS Warehouse B2B Enterprise UX

Role

  • UX/UI Design
  • Visual Design
  • Design System

Timeline

10 months · 2023–2024

Tools

  • Figma
  • Cloudscape Design System

System context

WEMALO WMS is the core warehouse management system of the 4elements Group ecosystem, used by logistics operators and fulfillment providers managing inventory for multiple stores.

  • Master data (users, clients, products, warehouses).
  • Operational flows (inbound, picking, packing, relocation, returns).
  • Administrative management and reporting (services, movements, KPIs).

It is the backbone of logistics operations: connecting physical tasks (scanning, packing, transport) with the digital visibility that administrators and clients need.

Design challenge

The system was powerful and feature-rich, but its interface was outdated and visually inconsistent.

  • Dense, rigid design that was hard to scan.
  • Weak information hierarchy in complex modules.
  • Different visual patterns across sections.
  • Navigation friction between operations and configuration.

Constraint

Modernize the system without changing the existing technical architecture (Bootstrap + DataTables), and improve the experience for both operational and admin users.

Redesign objectives

  • Improve visual clarity for dense data management.
  • Establish consistency across modules with a coherent visual system.
  • Lay a scalable foundation for future UI improvements without a full rebuild.
  • Adopt an enterprise approach inspired by Cloudscape (AWS), adapted to WEMALO’s brand identity (dark blue + orange).

My role & strategy

As the company’s sole UX/UI designer, I owned the entire visual redesign: from analysis to high-fidelity prototypes and design system decisions.

Instead of building a design system from scratch, I used Cloudscape Design System as structural base and adapted it to WEMALO.

  • Accelerated implementation with a mature enterprise foundation.
  • Maintained clarity and readability in dense interfaces.
  • Integrated WEMALO’s dark blue and orange to preserve visual DNA.

System analysis

Understanding the functional structure before changing visuals.

Functional groups

  • Stammdaten — master data for users, warehouses, clients and products.
  • Konfiguration — shipping profiles, packaging, data exchange and services.
  • WE (Goods In) — receiving, scanning, returns registration and quick entries.
  • Lagerverwaltung — relocations, inventory counts, refurbishment, stock state.
  • WA (Goods Out) — picking, packing, KO-box, transports, docks.
  • Berichte — movements, history, services, automated reports, serials.

Main conclusion

The system already covered the full logistics chain. The problem wasn’t functionality, it was visual design:

  • Too much information on flat, undifferentiated screens.
  • Little hierarchy between navigation, filters and content.
  • Inconsistent patterns across modules.

The redesign focused on hierarchy, consistency and visual scalability, without touching core logic.

Redesign process

From enterprise benchmarks to a scalable visual system.

1 · Benchmarking & references

I analyzed leading enterprise products for dense systems:

  • Cloudscape (AWS) — clarity and visual scalability.
  • SAP EWM — complex functional structures.
  • Odoo / Netsuite / Oracle WMS — modularity and hierarchies.

I chose Cloudscape as base for its readability and robustness, then adapted it to WEMALO’s brand.

2 · Visual language definition

I translated Cloudscape principles into a WEMALO-specific visual language:

  • Dark blue (trust) + orange (action) as core brand colors.
  • Neutral grays for background and table density.
  • Clear typographic hierarchy for titles and tables.
  • Separation between navigation, filters and content areas.

3 · Base component design

I built a Figma library of core components adapted to WMS needs:

  • Tables with sticky headers and hover states.
  • Tiered filters with active badges and saved views.
  • Contextual toolbars for repetitive actions.
  • Icons and indicators aligned with the color system.

4 · Prototyping & validation

I created high-fidelity prototypes for key modules (Inbound, Outbound, Inventory, Reports) and validated:

  • Technical compatibility with Bootstrap + DataTables.
  • Visual consistency across modules.
  • Documentation in Figma for future development.

Cloudscape design system adaptation

Using an enterprise system as foundation, not as a visual shortcut.

Main improvements

How the redesign transformed the visual layer without breaking the system.

Information density

  • Before: dense screens with little hierarchy.
  • After: modular layouts separating navigation, filters and tables.

Visual identity

  • Before: neutral colors without a clear brand presence.
  • After: dark blue + orange palette aligned with WEMALO.

Filters & search

  • Before: scattered filters across screens.
  • After: tiered filters with visible active badges and saved views.

Tables & actions

  • Before: hard-to-read tables with redundant actions.
  • After: optimized typography, spacing and contextual toolbars.

Learnings & metrics

What worked, trade-offs and how I’d evaluate this type of redesign.

Learnings & retrospective

  • Adopting Cloudscape as base gave speed and consistency without building a DS from zero.
  • The biggest UX gains came from hierarchy: filters + badges + saved views in dense tables.
  • Brand alignment over an enterprise base elevated perception without sacrificing readability.

Trade-offs & collaboration

  • Less visual freedom in exchange for mature components.
  • Time invested in tuning density and spacing for WMS contexts.
  • Walkthroughs with operations + quick technical validation worked better than long testing cycles.
  • Figma documentation of components, tokens and usage simplified dev handoff.

Metrics I’d measure

Quantitative metrics weren’t tracked at baseline, but for similar projects I’d measure:

  • Time to locate an order (p95).
  • Clicks/scrolls to apply critical filters.
  • Errors from incorrect state selection.
  • Onboarding time for new operators.

Expected impact: −20–35% search time, −15–25% errors, and ~−25% onboarding time with better Saved Views and guidance.

Open questions & design ops

  • Which saved views should be preconfigured per role (Admin, Operator, Client)?
  • Which KPIs are critical for day-to-day vs. month-end reporting?
  • When is the right moment to migrate to React without affecting peak operations?
  • Design tokens: color, spacing and typography defined for reuse.
  • Contribution rules for the Figma library (naming, variants, states).
  • Accessibility checklist (contrast, focus, keyboard states) per component.
