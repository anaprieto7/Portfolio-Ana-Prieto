WEMALO Dashboard — B2B Client Interface Redesign

WEMALO Dashboard is the B2B portal where warehouse clients manage their inventory, orders and returns within the WEMALO ecosystem. Unlike the internal WMS, this dashboard is built for merchants and warehouse managers who need full visibility without using the internal warehouse system.

Challenge & solution

From problem to solution

Restructuring the B2B client portal around roles, tasks and real decisions.

Overview

WEMALO Dashboard is the external interface for clients of the WEMALO ecosystem. It gives merchants and warehouse managers visibility over inventory, orders and returns — without exposing the internal WMS complexity.

The challenge was not to add more features, but to make the existing ones understandable and relevant for each actor interacting with the system.

Design challenge

Technically, the system worked — but its information architecture was confusing and poorly adapted to different user types.

  • Identical flows for all roles (SuperAdmin, Admin, Merchant).
  • Redundant information repeated across screens.
  • Navigation that didn’t reflect each actor’s operational context.
  • Cognitive overload and unnecessary management time.

Redesign objectives

Restructure IA and interaction logic to improve clarity, relevance and flow:

  • Tailor screens and flows to each user type’s needs.
  • Reduce noise and unnecessary steps.
  • Deliver a coherent, scalable, task-centered experience.

Roles & user profiles

Who we designed for

Clarifying mental models before redefining flows.

Admin

Warehouse manager responsible for logistics operations and relationships with assigned merchants.

  • Needs multi-client visibility.
  • Monitors SLAs, warehouse capacity and exceptions.
  • Prioritizes control and quick decisions.

Merchant

Client who manages their own inventory, creates inbound orders, and monitors orders and returns.

  • Needs clear views of stock, orders and returns.
  • Focuses on product health and end-customer impact.
  • Wants simple flows for inbound and returns.

Each role has different goals and information needs — the redesign aligns navigation, modules and actions with these mental models.

Process

From IA audit to role-based flows and UI refinements.

1 · Analysis

Full audit of existing views and permissions. I mapped which data and actions each role actually used day to day, and where they struggled to find information.

Output: roles/permissions matrix, list of redundant screens and critical flows.

2 · Information architecture

I reorganized the IA around tasks instead of database entities. New main navigation:

  • Dashboard
  • Orders
  • Inbound Orders
  • Inventory
  • Returns
  • Settings

Each module exposes contextual information and actions per role.

3 · Interaction design

I redesigned core flows (order creation, stock review, returns) with reusable patterns:

  • Clear primary vs secondary actions.
  • Consistent feedback and empty states.
  • Tables with action cells and status chips.

4 · UI design

I applied a Cloudscape-inspired UI with modular layouts, clear typography and dark-mode support, using the Vuexy template as a technical base for implementation.

  • Simplified cards and tables.
  • Sharper hierarchy in dashboard metrics.
  • Visual coherence with WEMALO WMS and Retoure24.

Information architecture & current state

Making sense of what existed before changing it.

Solution

Role-based dashboards, contextual modules and task-centered flows.

The new WEMALO Dashboard groups information and flows by user role, delivering a clear, task-oriented experience. SuperAdmins get global visibility, Admins manage multi-client operations, and Merchants control their products and inventory with ease.

  • Role-specific flows instead of one-size-fits-all.
  • Visual hierarchy optimized for daily operations.
  • Simplified, contextual navigation per user type.
  • Tables structured by decision priority (status, exceptions, next actions).
  • Clear visual feedback and direct actions on each row.

Key improvements

  • Dashboard cards per role (Admin vs Merchant) with relevant KPIs.
  • Orders and Inbound split clearly by responsibility and state.
  • Inventory views tailored to “what do I need to act on now?”.
  • Returns integrated with warehouse flows while staying understandable for merchants.

Before & after

From generic, flat screens to role-based dashboards and clean tables.

Impact

The redesign significantly improved clarity, efficiency and scalability of the WEMALO client portal.

  • ~45% reduction in average information search time.
  • Role-differentiated views, eliminating permission ambiguity.
  • Order creation flows reduced from 5 steps to 2.
  • Visual base consistent with other WEMALO ecosystem products.

Reflection

The most complex challenge was redesigning the invisible logic behind the screens. It wasn’t about colors or icons — it was about making sure each user understood the system through the information they truly need.

This project reinforced the importance of designing for roles and mental models, not for internal technical architecture. The same data can be arranged very differently depending on who is looking at it and why.

Next steps

How I would evolve WEMALO Dashboard from here.

  • Validate the prototype with real warehouse managers and merchants, focusing on navigation, dashboard comprehension and flows.
  • Integrate usage analytics (search terms, filters, path analysis) to iterate on high-impact modules.
  • Design advanced reporting views for warehouse KPIs and performance per client.

Design system perspective

The dashboard shares a visual language with WEMALO WMS and Retoure24, using Cloudscape as base. The next step is formalizing:

  • Shared tokens (color, spacing, typography).
  • Component variants for internal vs client-facing screens.
  • Guidelines for role-based dashboards across the ecosystem.
